Open-source plugin = local-only by design.

Context Mode operates at the MCP protocol layer — raw data (web pages, API responses, file analyses, log files) stays in a sandboxed subprocess on the developer machine and never enters the LLM context window. The SQLite knowledge base lives in the home directory and dies when the session ends.

Nothing leaves the machine. No telemetry, no cloud sync, no usage tracking, no account required. The 250,000+ developers running the plugin today have never sent a byte to us.

Insights = explicit opt-in. Generate a token in the dashboard, the same plugin starts forwarding structured events (tool name, file path, error count — never prompt content, never source code). Remove the token, forwarding stops.

§ 02 · the solution

Insights — the analytics layer on top of the open-source plugin.

The context-mode OSS plugin already runs locally for 250,000+ developers. Every Claude Code, Cursor, Copilot, Codex, Gemini, Kimi, Qwen, JetBrains, Kiro session emits structured events through its hook system. Insights ingests those events server-side and runs 220 patterns over them.

The findings surface in three places — same data, three doors in. Web Dashboard for the human eye. REST API for CI/CD and compliance pulls. Remote MCP for the AI agent reasoning over your engineering operations live.

Nothing on the developer laptop changes. The hook fires fire-and-forget on every tool call. Same plugin. Same workflow. Three new doors.

All operations are role-narrowed at the schema layer. A Member token cannot pass scope: "org" — the enum literally lacks the value. Defense at the schema, not the controller.

§ 06 · what we ingest

Structural events. Never source code.

Read this first: the OSS plugin ships nothing.

The open-source plugin you install runs 100% locally — FTS5 SQLite on your machine, zero telemetry, zero cloud calls, no account. 250,000+ developers use it today without ever sending us a byte.

The schema below describes Insights only — the opt-in org layer. The same plugin starts emitting these structured events only after you generate a token in the dashboard and paste it into your config. Remove the token, forwarding stops. No silent uploads, ever.

The open-source plugin emits a tightly-scoped envelope per AI tool call. Every field is observable to the local engineer — nothing surprising travels.

01 · Tool calls
What the agent did
  • Tool name (Read, Edit, Bash, Task, …)
  • Duration, exit status
  • File paths touched
  • Error / retry counts
02 · Session signal
How long, how deep
  • Session start / end
  • Compact cycles
  • Source files surfaced
  • Search queries (count, not text)
03 · Prompt features
10 universal signals
  • Unicode script (Latin / Han / Arabic / …)
  • Length, word count, uppercase ratio
  • Code-block count, URL count
  • Streaming word tokens (DF-gated)
04 · Git context
What shipped
  • Commit message subject (verb-family classified)
  • Worktree lifecycle
  • ExitPlanMode signals (plan bytes, hash)
  • Agent token usage + cost_usd

+ Structural events we ingest

  • Tool name, duration, exit status
  • File paths touched (basename, not content)
  • Error category enum (typed taxonomy)
  • Commit message subject only (`-m` argument)
  • Session metadata (start, compact count, source count)
  • 10 Unicode-property prompt features (per §11 Layer-1)
  • Streaming word tokens (deduped, lowercased, length ≥ 3)
  • Agent token usage + per-model cost derivation

− What we never collect

  • Prompt content (raw user text)
  • File content (any byte of any source file)
  • Tool output (response payloads stay local)
  • Authentication tokens / API keys
  • Personally identifiable information beyond org-set email
  • Per-language vocabulary lists or stop-word tables
  • Cross-org event leakage (member-scope strictly enforced)
  • Telemetry from the OSS plugin without explicit token grant

§ 10 · security & access

Three roles. One schema-enforced ceiling.

Most analytics products bolt RBAC on top with middleware checks. Insights enforces it one layer earlier — at the MCP tool schema itself.

owner
org
CTO, founder, billing owner. Sees everything. Manages members, tiers, billing. scope enum includes "org", "team", "member", "self".
manager
team-set
EM, Team Lead, Director, CISO, DevOps Lead. managed_teams column narrows the read — 1 team = Team Lead, N teams = EM. Server filters; schema enforces.
member

IC / developer. Sees own metrics plus anonymised peer benchmarks. Their team appears as "Team A" when peers compare. scope enum literally lacks "org".

Member-scope reads bypass org-aggregate whitelist filters — a member's full activity counts toward their personal KPIs even when their project is pending org review. Defense at the schema layer, not the controller.

§ 12 · faq

Questions before the call.

Is Insights live yet? How do I get in?

Private beta. Insights is onboarding 5 design-partner orgs first. Join the waitlist for early access — design-partner terms (free Team-tier access during the design phase) are discussed there. Pricing is intentionally not published yet — it will be shaped with the first cohort. The OSS plugin remains fully usable today, no waitlist, no account.

What data do you ingest?

Structured session events emitted by the OSS plugin's hook system — never prompt content, never source code, never file content. The envelope covers: tool calls (name, duration, exit status), prompt boundaries, file paths touched, error and retry counts, commit message subjects, agent token usage with derived cost, session metadata (compact count, sources indexed, search queries), ExitPlanMode signals (plan byte size, plan hash), ExitWorktree signals (discard-changes flag), SessionStart settings and MCP-server snapshots, Layer-3 streaming word-token counts, and 10 universal Unicode-property prompt features. The plugin operates locally; only the structured envelope ships when an Insights token is present.

Who sees what?

Owner sees the org. Manager sees the teams listed in their managed_teams column. Member sees themselves plus anonymised peer benchmarks. Enforced at the MCP schema layer — a Member token literally cannot pass scope: "org" because the enum lacks the value.

Does the OSS plugin still work without Insights?

Yes — and that is the default posture. The OSS plugin is licensed under Elastic License 2.0 (source-available) and operates at the MCP protocol layer: raw data stays in a sandboxed subprocess and never enters the LLM context window. Nothing leaves the machine — no telemetry, no cloud sync, no usage tracking, no account required. The SQLite knowledge base lives in the home directory and dies when the session ends. Insights is an explicit opt-in: generate a token in the dashboard and the same plugin starts forwarding structured events. Remove the token, forwarding stops.

How are the patterns built?

Empirically, from real session telemetry across the OSS adapter surface (15 platform integrations on the OSS side; 10 currently exposed in the Insights install CLI — Claude Code, Codex, Gemini, Qwen, Cursor, OpenCode, Kimi, VS Code Copilot, JetBrains Copilot, and Kiro). The catalog ships 219 patterns across 8 categories (adoption, effectiveness, efficiency, quality, workflow, engagement, prompt, security), each one wired through getMetricValue + AggregateData in the engine and independently testable. Every pattern is SQL + threshold + severity + suggested action, audited against existing event flows.

What is the agentic / MCP angle?

The remote MCP endpoint is a first-class surface — every dashboard widget has an MCP twin. The agent server ships 13 tools today: find_blockers, what_changed, top_patterns, how_is_team, compare_teams, how_am_i, identify_risk, engagement_health, top_members, activity_timeline, activity_breakdown, member_detail, and commit_quality_team. Tool schemas are dynamically computed per token — the schema itself enforces access boundaries (a Member token cannot ask org-scope questions because the enum lacks the value).

Does cross-language work?

Yes by construction. Commit Quality classifier uses Conv Commits ASCII prefix (cross-language) + structural signals (issue refs, file paths) — never English verb dictionaries. Prompt features use Unicode property regex (`\p{Script=X}`, `\p{L}+`) — never language detection libraries. Streaming topics use DF-ratio (count / N_prompts) which is organic to the org's actual vocabulary, Turkish or otherwise.

On-prem option?

Available for design partners on request. The stack runs on Cloudflare Workers + D1 + Durable Objects — deployable to any Cloudflare account the customer controls, with the D1 schema and pattern catalog open. Fully air-gapped self-hosted SOC2 / FedRAMP work is custom — join the waitlist for design-partner inquiries.

What other Solutions are on the roadmap?

Insights is the first Solution from Context Mode Platform. Two more on the roadmap: Audit turns the event stream into compliance reports for the CISO persona. Risk Score scores AI-authored changes inside a PR for the DevOps Lead persona, exposed as a GitHub Action. Neither ships in v1 — design partners welcome.

How does Insights relate to Context Mode Platform?

Context Mode Platform is the umbrella — the OSS plugin, the event ingestion layer, the pattern engine, the dashboard, the remote MCP. Insights is the first Solution built on top of it: the org-layer analytics surface, role-tuned, persona-aware. Future Solutions (Audit, Risk Score) ride the same data layer.
